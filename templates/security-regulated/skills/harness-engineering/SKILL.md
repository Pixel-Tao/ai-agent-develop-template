# Skill: Harness Engineering

## 1. 목적

하네스 엔지니어링은 프로젝트 작업을 실행 가능한 명령, 검증 기준, 증거 로그, 산출물 위치로 묶어 반복 가능하게 만드는 방법이다. Agent가 같은 작업을 다시 수행하거나 다른 담당자가 인수인계받아도 판단 근거를 추적할 수 있어야 한다.

## 2. 사용 조건

프로젝트 초기화, 구현 전 준비, 릴리즈 전 검증, 장애 대응, 보안 검토, 데이터/모델 평가처럼 실행 결과의 재현성이 중요한 작업에서 사용한다. 명령어가 아직 불명확한 기존 프로젝트에서는 먼저 후보 명령과 확인 질문을 분리한다.

## 3. 입력

- 작업 요청과 목표
- manifest.yaml과 AGENTS.md
- harness/harness.yaml
- 기존 명령, CI 설정, 테스트 방식
- 품질 게이트와 제약사항

## 4. 작업 절차

1. 작업 유형과 위험도를 분류한다.
2. commands.md에서 실행 가능한 명령을 찾는다.
3. 명령이 없으면 후보 명령과 확인 필요 사항을 기록한다.
4. verification-matrix.md에서 최소 검증을 선택한다.
5. 실행 결과와 증거 위치를 evidence-log.md에 기록한다.
6. 완료 보고에 검증 결과, 한계, 다음 조치를 포함한다.

## 5. 검토 기준

- 명령이 실제 프로젝트에서 실행 가능한가?
- 검증 기준이 작업 위험도에 맞는가?
- 실패 결과와 미검증 항목이 숨겨지지 않았는가?
- 산출물이 reports/ 또는 관련 문서에 남아 있는가?

## 6. 출력 형식

- Harness plan
- Command inventory update
- Verification matrix update
- Evidence log entry
- Risk or blocker note

## 7. 품질 기준

하네스는 최소한 실행 명령, 기대 결과, 실제 결과, 증거 위치, 남은 위험을 포함해야 한다. 자동 검증이 불가능한 경우 수동 검증 방법과 한계를 명시한다.

## 8. 금지사항

- 실행하지 않은 테스트를 통과한 것으로 보고하지 않는다.
- 실패 로그를 삭제하거나 요약 없이 숨기지 않는다.
- 위험 명령을 승인 없이 실행하지 않는다.
- 임시 명령을 공식 하네스로 확정하지 않는다.
