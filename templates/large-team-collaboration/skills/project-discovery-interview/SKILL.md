# Skill: Project Discovery Interview

## 1. 목적

사용자의 막연한 프로젝트 아이디어를 목표, 사용자, 콘텐츠, 기술 방향, 범위, 성공 기준이 있는 실행 가능한 프로젝트 정의로 구체화한다. 이 Skill은 구현을 서두르기 전에 무엇을 왜 만드는지 확인하는 안전장치다.

## 2. 사용 조건

프로젝트 초기화, 신규 기능 정의, 템플릿 적용, 요구사항이 모호한 작업에서 사용한다. 사용자가 "대충 이런 것"만 설명했거나 결과물의 성공 기준이 불명확하면 반드시 먼저 사용한다.

## 3. 입력

- 사용자의 최초 아이디어 또는 요청
- 프로젝트 유형 후보
- 제약사항, 선호 기술, 참고 자료
- 이미 존재하는 요구사항 또는 문서

## 4. 작업 절차

1. 사용자의 최초 설명을 원문에 가깝게 기록한다.
2. 목표, 사용자, 문제, 성공 기준을 질문한다.
3. 콘텐츠, 데이터, 화면, 워크플로우를 구체화한다.
4. 기술 스택, 외부 연동, 운영, 보안 제약을 확인한다.
5. 필수 범위, 다음 버전 범위, 제외 범위를 분리한다.
6. 결정, 가정, 미정을 표로 정리한다.
7. project-definition-brief를 작성한다.

## 5. 검토 기준

목표가 결과 중심으로 쓰였는지 확인한다. 사용자가 누구인지, 어떤 콘텐츠나 데이터를 다루는지, 무엇을 만들지와 만들지 않을지가 구분되어야 한다.

## 6. 출력 형식

- 인터뷰 요약
- Project Definition Brief
- 결정 목록
- 가정 목록
- 미정 질문 목록
- 다음 문서화 대상

## 7. 품질 기준

브리프만 읽고도 PM, 개발자, QA, AI Agent가 같은 프로젝트를 상상할 수 있어야 한다. 모호한 형용사보다 관찰 가능한 기준과 예시를 사용한다.

## 8. 금지사항

사용자의 첫 답변을 검증 없이 최종 요구사항으로 확정하지 않는다. 기술 스택을 목표보다 먼저 고정하지 않는다. 미정인 내용을 Agent가 임의로 채워 확정하지 않는다.

## 9. 초기 개발 문서 활용

사용자가 inputs/initial-development-docs/ 또는 inputs/references/에 문서를 넣어둔 경우, 이 Skill은 해당 문서를 먼저 읽고 source-documents-index.md에 근거를 정리한다. 문서에서 나온 요구사항 후보는 바로 확정하지 않고 인터뷰 질문으로 바꿔 사용자에게 확인한다.
