# Agent Instructions

## 역할

AI Agent는 프로젝트의 구조와 현재 상태를 먼저 파악한 뒤, 요청된 작업을 안전하고 유지보수 가능한 방식으로 수행한다. 사람과 협업하는 개발자처럼 영향 범위, 변경 이유, 검증 결과를 명확히 남긴다.

## 기본 원칙

기존 규칙과 컨벤션을 우선한다. 모르는 내용은 단정하지 않고 assumptions 또는 unknowns에 기록한다. 위험한 변경은 승인 조건을 명확히 표시한다.

## 우선 읽을 파일

1. README.md
2. manifest.yaml
3. AGENTS.md
4. docs/09_agent_state/current-status.md
5. 작업과 관련된 Skill 문서

## Skill 사용 규칙

작업 성격에 맞는 Skill을 선택하고, Skill의 입력/절차/품질 기준을 따른다. Skill이 맞지 않거나 정보가 부족하면 run log와 assumptions에 이유를 기록한다.

## 문서 작성 규칙

문서는 Markdown, 메타데이터는 YAML을 사용한다. 파일명은 lower-kebab-case를 사용하고 날짜는 YYYY-MM-DD 형식으로 기록한다.

## 코드 작업 규칙

변경 전 영향 범위를 확인한다. 기존 스타일과 테스트 구조를 보존하며, 관련 없는 리팩터링은 하지 않는다.

## 테스트 규칙

변경 범위에 맞는 가장 작은 테스트를 먼저 실행한다. 테스트가 없거나 실행할 수 없으면 이유와 남은 위험을 기록한다.

## 작업 로그 규칙

작업 시작, 주요 판단, 변경 파일, 검증 결과, 다음 작업을 docs/09_agent_state/run-log.md에 기록한다. 인수인계가 필요하면 handoff 문서도 갱신한다.

## 금지사항

승인 없이 삭제, 대규모 리팩터링, 공개 API 변경, 보안 정책 변경을 하지 않는다. 민감정보를 코드, 로그, 문서에 기록하지 않는다.

## 작업 완료 보고 형식

- 수행한 작업
- 변경한 파일
- 검증 결과
- 남은 위험 또는 확인 필요 사항
- 다음 권장 작업

## 프로젝트 발견 인터뷰 규칙

사용자의 요청이 대략적이거나 목표, 대상 사용자, 성공 기준, 기술 범위가 불명확하면 구현 전에 project-discovery-interview.md를 사용한다. Agent는 목표, 콘텐츠, 데이터, 기술 제약, 품질 기준, 제외 범위를 질문하고 project-definition-brief를 만든 뒤 다음 작업으로 넘어간다.
